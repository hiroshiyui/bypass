# Device identity key: Ed25519 in `$XDG_CONFIG_HOME/bypass/identity.key`

* Status: accepted
* Date: 2026-05-22
* Deciders: hiroshiyui

## Context and Problem Statement

Phase 5.2 pairing ([ADR-0012](0012-pake-spake2.md)) bootstraps
mutual trust via a one-shot PIN, then pins peer identities for
all subsequent connections. Each device therefore needs a
**stable libp2p identity keypair** that:

- persists across reboots,
- can be read back at daemon startup and at every pairing
  attempt,
- is the same key the *other* devices in the user's fleet have
  pinned in their `peers.toml`.

The eval doc flagged this as open question #6 (identity-key
location, perms, format, rotation). Sub-milestone 5.2.a needs
those answers before any persistence code lands.

## Considered Options

**Storage location:**

* `$XDG_CONFIG_HOME/bypass/identity.key` (alongside
  [`peers.toml`](0012-pake-spake2.md)). One config directory
  holds all per-device sync state.
* `$XDG_DATA_HOME/bypass/identity.key` — XDG-purist: the key is
  *data*, not config. Splits state across two directories;
  marginally more standards-compliant; more annoying to back up.
* Per-store `<store-root>/.bypass/identity.key`. Travels with
  the store. Couples identity to a single store and breaks the
  "one device syncs multiple stores" workflow.

**On-disk format:**

* libp2p protobuf (`identity::Keypair::to_protobuf_encoding`).
  Round-trips through libp2p; future-proofs algorithm migration
  because the encoding tags the curve.
* Raw 32-byte Ed25519 seed. Smallest; locks us to Ed25519
  forever.
* PEM / PKCS#8. Most interoperable with external tooling; nobody
  external needs to read this file.

**Algorithm:**

* Ed25519. libp2p's default; fast, small, modern, ubiquitous in
  every libp2p deployment we'd compare against.
* Secp256k1, RSA, ECDSA. No reason to prefer.

**Rotation:**

* Manual via `bypass sync identity rotate`. User triggers; old
  key destroyed; user re-pairs every device.
* Automatic on schedule. Premature; threat model doesn't justify.
* Defer rotation entirely. Risks shipping without a recovery
  story.

## Decision Outcome

- **Storage location:** `$XDG_CONFIG_HOME/bypass/identity.key`,
  with `0600` permissions on Unix (chmod the file at write time;
  refuse to load if perms are wider than `0600` to catch
  accidentally-leaked keys). `$XDG_CONFIG_HOME` defaults to
  `~/.config` on Linux and macOS when unset; Windows falls back
  to `dirs::config_dir()`.
- **On-disk format:** libp2p protobuf, via
  `identity::Keypair::to_protobuf_encoding` /
  `from_protobuf_encoding`. No human-readable headers; the file
  is raw bytes. (Operationally identical to how IPFS persists
  its peer identity.)
- **Algorithm:** Ed25519. Generated on first need
  (`bypass sync pair --show` / `--enter` if the file is absent;
  also auto-generated by `bypass sync daemon` startup in 5.2.c
  if it still isn't there).
- **Rotation:** explicit `bypass sync identity rotate`. The
  command:
  1. Refuses unless `--confirm` is also passed (rotation
     unconditionally invalidates every pinned peer relationship,
     so the user must opt in twice).
  2. Generates a fresh Ed25519 keypair.
  3. Overwrites `identity.key` atomically (tempfile + rename, mode
     `0600`).
  4. Clears `peers.toml` (every peer was pinning the *old* key;
     none of them are reachable post-rotation until re-paired).
  5. Prints a one-line "re-pair every device" reminder.

  Rotation is *not* a periodic-cron-style operation; it is the
  recovery path when the user believes the key was compromised
  (laptop stolen and recovered, suspected malware that read the
  file, ...). Phase 5.2 does not implement automatic rotation.

### Consequences

* Good: one config directory holds the sync identity + the
  pinned-peer table. Backup story is "copy
  `$XDG_CONFIG_HOME/bypass/`".
* Good: libp2p protobuf encoding means the file written by
  `bypass` can be loaded by *any* libp2p-based tool that
  happens to recognise the same protobuf. We don't anticipate
  that being useful day-one, but it leaves the door open for
  external migration / inspection tools.
* Good: Ed25519 gives 32-byte private keys, microsecond signs,
  no parameter choices to get wrong.
* Good: `0600` enforcement (both at write and at load) is the
  same posture `~/.ssh/id_ed25519` requires. Users with leaked
  perms get an immediate refusal rather than a silent risk.
* Bad: a per-store identity is less convenient for users who
  *do* run multiple unrelated stores on one device. The
  workaround (`PASSWORD_STORE_DIR=… BYPASS_IDENTITY=…`,
  selectable via env var) is reserved for a future ergonomics
  ADR if real users hit this.
* Bad: rotation is destructive — every paired peer relationship
  is lost. There is no "rotate but preserve trust" path because
  every peer pinned the *old* key. Mitigated by the
  `--confirm` double-opt-in.
* Bad: the protobuf format is libp2p-specific. If we ever swap
  libp2p (per ADR-0010's escape valve), the identity file may
  need a migration. Acceptable: that's a once-in-a-product
  cost, scoped to a future ADR.

### Confirmation

* Sub-milestone 5.2.a code MUST refuse to load `identity.key`
  if its permissions are wider than `0600` (Unix). Reviewers
  should reject any loader that silently chmods or skips the
  check.
* `bypass sync identity rotate` MUST require `--confirm`; a
  unit test asserts this. The same test asserts `peers.toml` is
  cleared as part of rotation.
* `identity.key` MUST be written atomically (tempfile in same
  dir + rename) so a crash mid-write can't leave a partial key.
  Same pattern as `StorageFs::write` for `.gpg` blobs.
* The eval doc's open question #6 is closed by this ADR.
